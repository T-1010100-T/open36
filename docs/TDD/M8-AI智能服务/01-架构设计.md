## 1. 架构概述

### 1.1 设计目标

M8 AI智能服务是一个独立的Python微服务，基于LangGraph构建多Agent系统，为管理员提供自然语言驱动的平台运营能力。

**核心设计原则**：

- **服务独立**：作为独立微服务部署，不侵入现有服务
- **API复用**：Agent工具层封装现有服务的内部API，不直接操作数据库
- **同步/异步双模式**：简单任务同步返回，复杂任务异步执行
- **可观测性**：完整的执行日志、工具调用记录、错误追踪

### 1.2 服务定位

```
┌─────────────────────────────────────────────────────────────────┐
│                        Kong API Gateway                          │
└──────────────────────────────┬──────────────────────────────────┘
                               │
        ┌──────────────────────┼──────────────────────┐
        │                      │                      │
   ┌────▼─────┐         ┌─────▼──────┐        ┌─────▼──────┐
   │ M3 帖子  │         │ M8 AI服务  │        │ HOJ 判题   │
   │ 服务     │ ◄────── │ (新增)     │ ─────► │ 系统       │
   └──────────┘  调用API  └────────────┘ 调用API └────────────┘
```

M8服务的特殊之处：它不是传统的CRUD服务，而是一个**AI编排服务**——接收自然语言输入，通过LLM理解意图，调用工具（即其他服务的API）完成任务。

### 1.3 架构优势

| 优势 | 说明 |
|------|------|
| **无侵入** | 不修改现有服务代码，仅调用现有API |
| **独立迭代** | Agent逻辑、Prompt、工具可独立优化 |
| **技术异构** | Python + LangGraph，与Django服务解耦 |
| **可扩展** | 新增Agent只需注册新工具和路由规则 |

## 2. 多Agent架构

### 2.1 Agent拓扑

```
                    ┌─────────────────────┐
                    │     Admin 用户       │
                    └──────────┬──────────┘
                               │ 自然语言指令
                               ▼
                    ┌─────────────────────┐
                    │   Router Agent      │
                    │   (主Agent)         │
                    │                     │
                    │   职责：            │
                    │   - 意图识别        │
                    │   - 任务分发        │
                    │   - 结果汇总        │
                    │   - 简单查询直接处理 │
                    └───┬─────────────┬───┘
                        │             │
           ┌────────────▼──┐   ┌──────▼───────────┐
           │  Forum Agent  │   │  Problem Agent   │
           │  (论坛Agent)  │   │  (出题Agent)     │
           │               │   │                  │
           │  工具：        │   │  工具：           │
           │  - create_post│   │  - create_problem│
           │  - list_posts │   │  - gen_examples  │
           │  - update_post│   │  - gen_testcases │
           │  - list_sections│ │  - gen_solution  │
           └───────────────┘   └──────────────────┘
```

### 2.2 Agent职责定义

#### Router Agent（主Agent）

**角色**：意图识别 + 任务调度
**模型**：Claude Sonnet（性价比高，响应快）
**职责**：

1. 接收管理员自然语言输入
2. 分类意图（论坛/出题/查询/模糊）
3. 简单查询直接处理（如"查看板块列表"）
4. 复杂任务分发给子Agent
5. 汇总子Agent执行结果，返回给管理员

**路由逻辑**：

```python
def route_intent(user_message: str) -> str:
    """意图分类 → 路由到对应Agent"""
    # LLM判断意图类别
    intent = classify_intent(user_message)

    if intent == "forum":
        return "forum_agent"
    elif intent == "problem":
        return "problem_agent"
    elif intent == "query":
        return "direct_handle"  # 主Agent直接处理
    else:
        return "ask_clarification"  # 请求澄清
```

#### Forum Agent（论坛Agent）

**角色**：论坛内容管理
**模型**：Claude Sonnet
**工具集**：

| 工具 | 输入 | 输出 | 调用API |
|------|------|------|---------|
| `create_post` | title, content, section_id | post_id | `POST /internal/posts/` |
| `update_post` | post_id, title?, content? | success | `PUT /internal/posts/{id}/` |
| `list_sections` | 无 | sections[] | `GET /internal/sections/` |
| `list_posts` | section_id?, page? | posts[] | `GET /internal/posts/` |

**执行流程**：

```
1. 接收任务（如"在技术分享板块发一篇Git教程"）
2. 调用list_sections获取板块列表
3. 匹配"技术分享"板块的section_id
4. LLM生成帖子标题和Markdown内容
5. 调用create_post创建帖子
6. 返回执行结果
```

#### Problem Agent（出题Agent）

**角色**：算法题生成
**模型**：Claude Sonnet
**工具集**：

| 工具 | 输入 | 输出 | 说明 |
|------|------|------|------|
| `create_problem` | 完整题目数据 | problem_id | 调用HOJ API |
| `generate_examples` | 题目描述 | examples[] | LLM生成 |
| `generate_testcases` | 题目描述, 难度 | testcases[] | LLM生成 |
| `generate_solution` | 题目描述 | code | LLM生成参考解 |

**执行流程**：

```
1. 接收任务（如"出一道二分查找的简单题"）
2. LLM生成题目描述（title, description, input/output格式）
3. 调用generate_examples生成示例
4. 调用generate_solution生成参考解
5. 调用generate_testcases生成隐藏测试用例
6. 用参考解验证所有测试用例的正确性
7. 调用create_problem提交到HOJ
8. 返回执行结果
```

### 2.3 LangGraph状态图

```python
from langgraph.graph import StateGraph, MessagesState

class AgentState(TypedDict):
    messages: list          # 对话历史
    intent: str             # 识别的意图
    task_result: dict       # 任务执行结果
    error: str | None       # 错误信息

# 构建状态图
graph = StateGraph(AgentState)

# 添加节点
graph.add_node("router", router_agent)
graph.add_node("forum_agent", forum_agent)
graph.add_node("problem_agent", problem_agent)
graph.add_node("direct_handler", direct_handler)

# 添加边（路由逻辑）
graph.add_edge("router", route_decision)  # 条件路由
graph.add_conditional_edges(
    "route_decision",
    lambda state: state["intent"],
    {
        "forum": "forum_agent",
        "problem": "problem_agent",
        "query": "direct_handler",
    }
)

# 子Agent完成后 → END
graph.add_edge("forum_agent", END)
graph.add_edge("problem_agent", END)
graph.add_edge("direct_handler", END)

# 入口
graph.set_entry_point("router")
```

### 2.4 工具执行机制

每个Agent工具本质上是对后端API的封装：

```python
from langchain_core.tools import tool

@tool
def create_post(title: str, content: str, section_id: int) -> dict:
    """创建论坛帖子"""
    response = requests.post(
        f"{CONTENT_SERVICE_URL}/internal/posts/",
        json={"title": title, "content": content, "section_id": section_id},
        headers={"X-Internal-API-Key": INTERNAL_API_KEY}
    )
    return response.json()
```

**工具执行安全**：

- 所有工具调用通过内部API，不直接操作数据库
- 内部API Key存储在环境变量中
- 工具调用超时设置（单次30s）
- 错误隔离：单个工具失败不影响其他工具

#### 3.1.4 内部API调用模式

所有Python服务的内部API统一使用以下模式：

```python
import httpx

async def call_internal_api(service_url: str, method: str, path: str, **kwargs):
    """调用内部API的统一封装"""
    async with httpx.AsyncClient() as client:
        response = await client.request(
            method=method,
            url=f"{service_url}{path}",
            headers={"X-Internal-API-Key": settings.INTERNAL_API_KEY},
            timeout=30.0,
            **kwargs
        )
        response.raise_for_status()
        return response.json()
```

### 3.2 与HOJ系统集成

**调用方式**：HTTP REST（HOJ管理API）
**HOJ后端地址**：`http://hoj-backend:6688`（Docker网络） / `http://localhost:6688`（本地开发）
**数据库**：HOJ使用独立的MySQL实例，与Open436的PostgreSQL完全分离

**需要的API**（`Open436-hoj/hoj-springboot/DataBackup/.../controller/admin/AdminProblemController.java`）：

| API | 方法 | 用途 | 说明 |
|-----|------|------|------|
| `/api/admin/problem` | POST | 创建题目 | 需要Shiro `admin` 角色 |
| `/api/admin/problem` | PUT | 更新题目 | 编辑已有题目 |
| `/api/admin/problem` | GET | 查询题目列表 | 分页查询 |
| `/api/admin/problem/testcase` | POST | 上传测试用例 | 配合题目创建 |

**鉴权方式**：Apache Shiro Session（非JWT）

> ⚠️ HOJ使用Shiro鉴权，M8需要先调用HOJ的登录接口获取Session Cookie，然后在后续请求中携带。

**集成方案**：

1. M8服务启动时，使用HOJ管理员账号调用 `/api/login` 获取Shiro Session
2. Session Cookie缓存在Redis中，过期前自动刷新
3. 所有HOJ API调用携带此Session Cookie
4. HOJ管理员账号通过环境变量 `HOJ_ADMIN_USER` / `HOJ_ADMIN_PASS` 配置

### 3.3 与Kong网关集成

**Kong插件**：项目使用自定义 `satoken-auth` 插件（`deploy/dev/kong/plugins/satoken-auth/`）

**插件工作流程**：
1. 从 `Authorization` Header 提取Token（自动剥离 `Bearer ` 前缀）
2. 调用 Auth 服务 `POST /api/auth/verify` 验证Token
3. 验证成功后注入以下Header到上游请求：

| Header | 说明 | 示例 |
|--------|------|------|
| `X-User-Id` | 用户ID | `1` |
| `X-Username` | 用户名 | `admin` |
| `X-User-Role` | 角色代码 | `admin` |
| `X-User-Status` | 账号状态 | `active` |

4. 验证失败返回 401，Auth服务不可达返回 503

**M8需要在Kong中注册路由**（通过Kong Admin API `http://localhost:8001`）：

```
/api/ai/*  →  M8 AI服务 (端口 8008)
```

**路由规则**：

| 路径 | 方法 | 说明 | 鉴权 |
|------|------|------|------|
| `/api/ai/chat` | POST | 同步对话 | satoken-auth（仅admin） |
| `/api/ai/tasks` | POST | 创建异步任务 | satoken-auth（仅admin） |
| `/api/ai/tasks` | GET | 查询任务列表 | satoken-auth（仅admin） |
| `/api/ai/tasks/{id}` | GET | 查询任务状态 | satoken-auth（仅admin） |
| `/api/ai/tasks/{id}` | DELETE | 取消任务 | satoken-auth（仅admin） |
| `/api/ai/conversations` | GET | 会话列表 | satoken-auth（仅admin） |
| `/api/ai/conversations/{id}` | GET | 会话详情 | satoken-auth（仅admin） |
| `/api/ai/conversations/{id}` | DELETE | 删除会话 | satoken-auth（仅admin） |
| `/api/ai/agents` | GET | Agent列表 | satoken-auth（仅admin） |
| `/api/ai/agents/{name}` | PUT | 更新Agent配置 | satoken-auth（仅admin） |

**Kong路由注册方式**（在部署脚本中执行）：

```bash
# 创建服务
curl -X POST http://localhost:8001/services/ \
  --data "name=ai-service" \
  --data "url=http://ai-service:8008"

# 创建路由
curl -X POST http://localhost:8001/services/ai-service/routes \
  --data "name=ai-route" \
  --data "paths[]=/api/ai" \
  --data "strip_path=false"

# 启用 satoken-auth 插件
curl -X POST http://localhost:8001/services/ai-service/plugins \
  --data "name=satoken-auth" \
  --data "config.auth_service_url=http://auth:8081"
```

### 3.4 服务间通信时序

#### 同步对话流程

```
Admin UI → Kong → M8 AI服务 → LLM API
                         ↓
                   Router Agent (意图识别)
                         ↓
                   Forum Agent / Problem Agent
                         ↓
              M3/HOJ内部API (创建帖子/题目)
                         ↓
                   返回执行结果 → Admin UI
```

#### 异步任务流程

```
Admin UI → Kong → M8 AI服务 → 创建任务(PENDING) → 返回task_id
                         ↓
                   Celery Worker 异步执行
                         ↓
                   Agent执行 → M3/HOJ API
                         ↓
                   更新任务状态(COMPLETED/FAILED)
                         ↓
Admin UI → 轮询 task_id → 获取结果
```

## 4. 技术选型

### 4.1 核心技术栈

| 组件 | 技术选型 | 版本 | 选型理由 |
|------|---------|------|---------|
| **Web框架** | FastAPI | 0.115+ | 异步原生，Python生态，自动OpenAPI文档 |
| **Agent框架** | LangGraph | 0.3+ | 状态图驱动，支持条件路由、并行执行 |
| **LLM SDK** | langchain-anthropic | 0.3+ | Claude tool_use原生支持 |
| **任务队列** | Celery + Redis | 5.4+ | 成熟的异步任务框架 |
| **数据库** | PostgreSQL | 14+ | 复用现有实例 |
| **缓存** | Redis | 7+ | 复用现有实例（Celery Broker + 缓存） |
| **ORM** | SQLAlchemy | 2.0+ | 与现有Python服务一致 |
| **HTTP客户端** | httpx | 0.27+ | 异步HTTP，调用内部API |

### 4.2 LLM选型

| 场景 | 推荐模型 | 理由 |
|------|---------|------|
| Router Agent（意图识别） | Claude Sonnet | 快速、准确、成本适中 |
| Forum Agent（内容生成） | Claude Sonnet | 中文写作质量高 |
| Problem Agent（出题） | Claude Sonnet | 代码生成能力强 |

**备选方案**：OpenAI GPT-4o（通过LangChain统一接口切换）

## 5. 部署方案

### 5.1 Docker配置

在现有 `docker-compose.yml` 中新增以下服务（与现有服务保持一致的配置风格）：

```yaml
  # AI服务
  ai-service:
    build: ./Open436-AIService
    container_name: open436-ai-service
    restart: unless-stopped
    ports: ["8008:8008"]
    depends_on:
      postgres: { condition: service_healthy }
      redis: { condition: service_started }
    networks: [open436-net]
    environment:
      DATABASE_URL: postgresql://${PG_USER:-open436}:${PG_PASS:-open436}@postgres:5432/${PG_DB:-open436}
      REDIS_URL: redis://redis:6379/2
      ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}
      CONTENT_SERVICE_URL: http://content-service:8003
      SECTION_SERVICE_URL: http://section-service:8005
      USER_SERVICE_URL: http://user-service:8002
      HOJ_API_URL: http://hoj-backend:6688
      HOJ_ADMIN_USER: ${HOJ_ADMIN_USER:-admin}
      HOJ_ADMIN_PASS: ${HOJ_ADMIN_PASS:-admin123}
      INTERNAL_API_KEY: ${INTERNAL_API_KEY:-open436-internal-secret}
      CONSUL_URL: http://consul:8500
      CONSUL_SERVICE_NAME: ai-service
      SERVICE_PORT: "8008"
      APP_DEBUG: "false"
      APP_LOG_LEVEL: info

  # Celery Worker
  ai-worker:
    build: ./Open436-AIService
    container_name: open436-ai-worker
    restart: unless-stopped
    command: celery -A app.celery_app worker --loglevel=info --concurrency=2
    depends_on:
      postgres: { condition: service_healthy }
      redis: { condition: service_started }
    networks: [open436-net]
    environment:
      DATABASE_URL: postgresql://${PG_USER:-open436}:${PG_PASS:-open436}@postgres:5432/${PG_DB:-open436}
      REDIS_URL: redis://redis:6379/2
      ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}
      CONTENT_SERVICE_URL: http://content-service:8003
      SECTION_SERVICE_URL: http://section-service:8005
      USER_SERVICE_URL: http://user-service:8002
      HOJ_API_URL: http://hoj-backend:6688
      INTERNAL_API_KEY: ${INTERNAL_API_KEY:-open436-internal-secret}
```

> ⚠️ 需要在根目录 `.env` 文件中添加 `ANTHROPIC_API_KEY` 和 `INTERNAL_API_KEY` 配置。
